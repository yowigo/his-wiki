## 维度说明

### 副作用类型

| 标签 | 含义 | 可以随时用？ |
|------|------|------------|
| **只读** | 不修改任何文件，纯分析 | 随时，零风险 |
| **写文档** | 只写 .md 等文档文件 | 放心，不影响代码 |
| **写代码** | 生成新代码文件 | 注意，需要审查输出 |
| **修改代码+提交** | 改源码并 git commit | 谨慎，不可轻易撤销 |
| **提交+推送+PR** | 影响远程仓库 | 高风险，需确认 |
| **合并+部署** | 影响生产环境 | 最高风险，需确认 |
| **系统配置** | 修改会话级别的行为设置 | 低风险，可随时解除 |
| **系统操作** | 启动进程、执行升级等 | 中等风险 |

### 自动化程度

| 标签 | 含义 |
|------|------|
| **全自动** | 一键触发，中途不需要回答问题 |
| **半自动** | 关键节点暂停确认，其余自动 |
| **问答式** | 全程交互，Claude 提问你回答 |
| **拦截式** | 被动触发，当危险操作发生时介入 |

## 项目阶段地图

```
0→1 阶段（从想法到第一个可用版本）
─────────────────────────────────────────────
必用：/office-hours → /plan-ceo-review
      /plan-eng-review → /plan-design-review
      /design-consultation → /design-shotgun → /design-html
      /setup-deploy（一次性配置）
可选：/autoplan（替代三个 plan 技能）

1→N 阶段（功能迭代开发期）
─────────────────────────────────────────────
每个功能：/plan-eng-review → 写代码 → /design-html
          → /qa → /review → /ship → /land-and-deploy
持续：/benchmark（每次 PR）, /canary（每次部署）
调试：/investigate（遇到 Bug）
按需：/design-review（视觉打磨）, /codex（第二意见）

维护期（产品稳定，少量更新）
─────────────────────────────────────────────
常规：/retro（每周）, /cso（每月）
发布：/review → /ship → /land-and-deploy → /document-release
监控：/canary, /benchmark

紧急响应（线上故障）
─────────────────────────────────────────────
第一步：/investigate（找根因）
操作保护：/careful 或 /guard（防止误操作扩大故障）
验证修复：/browse → /qa-only → /canary
```

## 使用节奏日历

```
每次写代码时
  └─ /browse（随时验证页面状态）

每次准备提交 / 创建 PR 前
  └─ /review（代码审查）
  └─ /benchmark（性能对比）
  └─ /qa 或 /qa-only（功能验证）

每次发布
  └─ /ship（创建 PR）
  └─ /land-and-deploy（合并部署）
  └─ /canary（部署后监控 30 分钟）
  └─ /document-release（更新文档）

每次大功能开始前
  └─ /plan-eng-review（锁定架构）
  └─ /plan-design-review（确认视觉方案）

每周
  └─ /retro（回顾本周，追踪趋势）
  └─ /learn（查看积累的 learnings）

每月
  └─ /cso（安全审计）
  └─ /gstack-upgrade（更新 gstack）

项目初始化（一次性）
  └─ /setup-deploy（配置部署）
  └─ /design-consultation（建立设计系统）
  └─ /setup-browser-cookies（导入认证 Cookie）
```

## 决策 / 评审角色

---

### `/office-hours` — YC 风格 Office Hours

| 维度 | 值 |
|------|---|
| 副作用 | 只读 + 写文档 |
| 自动化程度 | 问答式（Claude 提六个强迫性问题） |
| 项目阶段 | 0→1（代码之前） |
| 使用节奏 | 按需，产品想法萌发时 |
| 输入 | 用户口头描述的想法 |
| 输出物 | 设计文档（保存到项目目录） |
| 跳过风险 | 做出错误的产品假设，在错误方向上浪费工程资源 |

**唤起时机**
- 描述一个新产品想法，想知道"这值得做吗"
- "帮我头脑风暴"、"我有个想法"、"想清楚这个方向"
- 项目还没有任何代码，处于概念探索阶段
- 黑客松、副项目、开源项目早期设计讨论
- 在 `/plan-ceo-review` 或 `/plan-eng-review` 之前使用

**职责**
- **创业模式**：六个强迫性问题——需求真实性、现状替代方案、极度细分的用户、最窄切入口、一手观察、长期适配性
- **建设者模式**：设计思维头脑风暴，适合副项目、学习型项目
- 输出设计文档，保存为项目档案

**边界**
- 不写代码，不做技术选型
- 不进入架构设计或实现细节
- 仅在"代码之前"阶段发挥作用

---

### `/plan-ceo-review` — CEO / 创始人视角计划评审

| 维度 | 值 |
|------|---|
| 副作用 | 只读 + 写文档 |
| 自动化程度 | 问答式（逐步审视计划，提出挑战） |
| 项目阶段 | 0→1、1→N 新功能立项时 |
| 使用节奏 | 每次重大决策或范围调整时 |
| 输入 | 现有计划文档 / 口头描述的方向 |
| 输出物 | 经过范围重新定义的计划文档 |
| 跳过风险 | 以小格局执行本可更大影响力的产品，战略失误代价高 |

**唤起时机**
- "想更大胆一点"、"扩大范围"、"战略层面评审"、"重新想这个"
- 现有计划感觉太保守，需要从顶层重新审视
- 在质疑计划的野心或边界感时

**职责**
- 从 CEO/创始人角度重新审视问题本身，寻找"10星级产品"版本
- 四种模式：**范围扩张** / **选择性扩张** / **锁定范围** / **缩减范围**
- 挑战计划的既有前提，提出结构性改变建议

**边界**
- 不做技术架构设计（交给 `/plan-eng-review`）
- 不做 UI/UX 评审（交给 `/plan-design-review`）
- 不负责执行计划的落地细节

---

### `/plan-eng-review` — 工程经理视角计划评审

| 维度 | 值 |
|------|---|
| 副作用 | 只读 + 写文档 |
| 自动化程度 | 问答式（逐个架构问题互动讨论） |
| 项目阶段 | 0→1、1→N 每个大功能开始前 |
| 使用节奏 | 每次开始实现前 |
| 输入 | 计划文档 / 功能描述 |
| 输出物 | 含架构图、数据流、边界情况的工程计划 |
| 跳过风险 | 架构问题在编码后才暴露，重构代价是编码前修改的 10 倍 |

**唤起时机**
- 有了计划或设计文档，准备开始写代码
- "审查架构"、"工程评审"、"锁定执行计划"

**职责**
- 锁定执行计划：架构选型、数据流、时序图
- 识别边界情况、测试覆盖缺口、性能隐患
- 以工程经理视角交互式走查，给出有明确立场的建议

**边界**
- 不做产品策略或范围决策（交给 `/plan-ceo-review`）
- 不做 UI/UX 评审（交给 `/plan-design-review`）
- 不负责实际写代码或提交变更

---

### `/plan-design-review` — 设计师视角计划评审

| 维度 | 值 |
|------|---|
| 副作用 | 只读 + 写文档 |
| 自动化程度 | 问答式（每个设计维度评分并互动改进） |
| 项目阶段 | 0→1、1→N 实现前 |
| 使用节奏 | 每次有 UI 组件的功能开始前 |
| 输入 | 含 UI/UX 描述的计划文档 |
| 输出物 | 各维度评分达到目标分数的改进计划 |
| 跳过风险 | 实现后才发现设计逻辑有问题，改设计比改代码还贵 |

**唤起时机**
- 有包含 UI/UX 组件的计划，准备实现前需要设计评审
- "审查设计计划"、"设计批评"

**职责**
- 对计划中每个设计维度评分 0-10，解释达到 10 分的条件
- 修改计划使其达到目标设计标准

**边界**
- 针对**计划文档**，不是已上线的页面（线上视觉审计请用 `/design-review`）
- 不做产品战略决策，不写实现代码

---

### `/autoplan` — 三评审联合流水线

| 维度 | 值 |
|------|---|
| 副作用 | 只读 + 写文档 |
| 自动化程度 | 全自动（用 6 条原则自动决策，最终汇总品味类问题） |
| 项目阶段 | 0→1、1→N |
| 使用节奏 | 有完整计划文件且不想逐步回答问题时 |
| 输入 | 计划文件 |
| 输出物 | 经过 CEO + 设计 + 工程三重评审的完整计划 |
| 跳过风险 | 同时跳过三个评审的叠加风险（战略失误 + 设计缺陷 + 架构问题） |

**唤起时机**
- 有完整计划文件，想一次性跑完所有评审
- "自动评审"、"帮我做决定"、"跑全部评审"
- 不想回答 15-30 个中间问题

**职责**
- 按顺序自动执行 CEO、设计、工程三个评审
- 用 6 条决策原则自动处理中间选择
- 将品味类、边界类决策集中在最终审批门呈现

**边界**
- 品味类、高风险类决策仍需用户最终确认
- 建议在 `/office-hours` 之后使用

### `/design-consultation` — 设计顾问

| 维度 | 值 |
|------|---|
| 副作用 | 写文档（DESIGN.md + 预览页 HTML） |
| 自动化程度 | 问答式（理解产品后提方案） |
| 项目阶段 | 0→1（项目初始化） |
| 使用节奏 | **一次性**，项目启动时 |
| 输入 | 产品描述 |
| 输出物 | DESIGN.md、字体+颜色预览页 |
| 跳过风险 | 无设计基准，每次 UI 决策各自为政，产品视觉风格混乱 |

**唤起时机**
- 启动全新项目的 UI，还没有任何设计系统或 DESIGN.md
- "设计系统"、"品牌规范"、"创建 DESIGN.md"

**职责**
- 深入理解产品定位，研究设计领域现状
- 提出完整设计系统：美学风格、字体、颜色、布局、间距、动效
- 创建 `DESIGN.md` 作为项目的设计基准文档

**边界**
- 不写功能代码，不负责将设计转化为实际页面（交给 `/design-html`）
- 对于已有站点，不推断既有设计系统（请用 `/plan-design-review`）

---

### `/design-shotgun` — 多变体设计探索

| 维度 | 值 |
|------|---|
| 副作用 | 写文件（多个 HTML 变体 + 对比看板） |
| 自动化程度 | 问答式（生成 → 反馈 → 迭代） |
| 项目阶段 | 0→1、1→N（任何视觉决策点） |
| 使用节奏 | 按需，UI 方向不确定时 |
| 输入 | UI 功能描述 / 现有页面 |
| 输出物 | 多个设计变体 HTML、结构化反馈记录 |
| 跳过风险 | 直接实现第一个设计方案，错过更好选择，返工成本高 |

**唤起时机**
- 描述了一个 UI 功能但还没看到实际长相
- "探索设计"、"给我看几个方案"、"设计变体"、"我不喜欢现在的样子"

**职责**
- 生成多个 AI 设计变体，打开对比看板
- 收集结构化反馈，根据反馈迭代直到确定方向

**边界**
- 产出是设计稿/变体，不是生产代码
- 不负责将确定的设计转化为正式代码（交给 `/design-html`）

---

### `/design-html` — 设计转生产代码

| 维度 | 值 |
|------|---|
| 副作用 | 写代码（生产质量 HTML/CSS 文件） |
| 自动化程度 | 全自动（方案已确定，无需中间决策） |
| 项目阶段 | 1→N（每个设计确定后） |
| 使用节奏 | 每次设计方案批准后 |
| 输入 | 批准的设计稿 / 设计文档 / 口头描述 |
| 输出物 | 生产质量 HTML/CSS（Pretext 原生，30KB，零依赖） |
| 跳过风险 | 手写 HTML 效率低，且容易偏离设计基准 |

**唤起时机**
- 已批准某个设计方案（来自 `/design-shotgun`、plan 评审或直接描述）
- "把设计转成 HTML"、"实现这个设计"、"做成页面"

**职责**
- 生成文字真实回流、高度动态计算、布局自适应的生产级 HTML/CSS
- 智能路由：根据设计类型选择合适的 Pretext 模式

**边界**
- 仅负责将已批准的设计转化为代码，不做设计决策
- 不负责后端逻辑或数据集成

### `/design-review` — 设计师视角视觉 QA

| 维度 | 值 |
|------|---|
| 副作用 | 修改代码 + 提交（每个修复原子提交） |
| 自动化程度 | 半自动（发现问题 → 修复 → 截图验证，可能询问优先级） |
| 项目阶段 | 1→N、维护期 |
| 使用节奏 | 每次迭代后视觉打磨时 |
| 输入 | 运行中的 Web 页面 URL |
| 输出物 | 修复前后对比截图 + git commits |

## 推荐工作流

### 完整项目生命周期

```
【0→1 阶段】
/office-hours          ← 想清楚做什么
    ↓
/autoplan              ← CEO + 设计 + 工程三评审（或分开跑三个 plan 技能）
    ↓
/design-consultation   ← 建立设计系统（一次性）
/setup-deploy          ← 配置部署（一次性）
    ↓
【1→N 迭代】
/plan-eng-review       ← 每个大功能开始前锁定架构
    ↓
写代码 + /browse       ← 随时验证页面
    ↓
/design-shotgun        ← 视觉方案探索（如有 UI）
/design-html           ← 设计转生产代码
    ↓
/qa                    ← 功能 QA + 自动修复
/design-review         ← 视觉 QA + 自动修复
    ↓
/review                ← 代码审查（只读）
/benchmark             ← 性能基线对比（只读）
    ↓
/ship                  ← 创建 PR
/land-and-deploy       ← 合并 + 部署
    ↓
/canary                ← 部署后监控 30 分钟
/document-release      ← 更新项目文档
    ↓
【维护期节奏】
每周: /retro           ← 工程复盘
每月: /cso             ← 安全审计
      /gstack-upgrade  ← 升级 gstack
按需: /learn           ← 回顾跨会话经验
```

### 紧急故障响应

```
发现故障
    ↓
/careful 或 /guard    ← 开启安全保护，防止误操作
    ↓
/investigate          ← 系统化找根因（铁律：无根因不动代码）
    ↓
修复代码
    ↓
/browse → /qa-only    ← 验证修复（只读，不引入新变更）
    ↓
/ship → /land-and-deploy → /canary
```
